# Lean cycle-worker agent + per-project cycle in task-loop.md — Implementation Plan

**Goal:** Move the step-by-step cycle out of the `cycle-worker` agent into the per-project
`docs/task-loop/task-loop.md` (generated by `create-cycle`, tailored per project). The agent keeps
only general operating principles + non-negotiable identity invariants and **follows `task-loop.md`
strictly**.

**Owner decisions (this thread, FINAL):**
- The cycle is **generated per project** by `create-cycle` (not a plugin-bundled reference).
- It lives in the **existing `task-loop.md`** (one project file: tailored cycle + parameters), not a
  new `cycle.md`.
- Every `cycle-worker` **follows `task-loop.md` strictly**.
- **Dead simple, no version gate.** No `PROTOCOL_VERSION`, no stamp, no preflight gate. Drift on a
  plugin upgrade is handled by a **documented manual step**: re-run `create-cycle` to regenerate
  `task-loop.md`. This means **no `.py` changes** — the change is purely doc/skeleton restructuring.
- The control/recovery JSON shapes move **as-is** (hand-emitted via `gh issue comment --body-file`,
  exactly as today) into `task-loop.md`; the `control_log`-formatter mandate is dropped to keep it
  simple.

**Codex-derived guardrails kept (cheap prose, not "machinery"):** the lean agent still carries the
non-negotiable identity invariants, a fail-closed boot ("read `task-loop.md` before acting; if
missing/unreadable, stop and ask the orchestrator"), and the precedence rule that prevents
`directions.md` steering from overriding protocol.

## Task 1: cycle-worker.md → lean core

Keep ONLY:
- **Identity + zero-side-effect boot:** do no git/GitHub/repo mutation before reading
  `docs/task-loop/task-loop.md`; if it is missing/unreadable or its `protocol_version` is
  incompatible with the installed plugin, **stop and ask the orchestrator** (no fallback copy, no
  acting from memory).
- **Compressed non-negotiable invariants** (must not depend on having read the cycle): never
  `gh pr merge`; never edit `proposal.md` (post `PLAN_FINDING`); write only your own worktree +
  per-attempt branch; before every irreversible boundary confirm `spawned_plan_revision` current
  AND `attempt_id == current_attempt_id`, else post the one terminal recovery status and stop; once
  a merge request starts, freeze the branch and bind `merge_request_uuid` to one head SHA.
- **General operating principles** (the owner wants these centralized): deliberate-with-codex (don't
  ask the user), tests-first, evidence-before-done, honor contracts, one reviewable PR per task,
  background long jobs, use available compute.
- **Inputs list + precedence rule:** `identity invariants > task-loop.md cycle/protocol >
  directions.md steering > task-loop.md params > issue text`. (Correct the current "directions.md is
  the highest-priority input of all" wording.)
- **Directive:** "Follow `docs/task-loop/task-loop.md` strictly, step by step."

Remove from the agent (moves to task-loop.md): the 11 steps, the worktree-setup bash, the
recovery-comment section, the control-protocol-events section, the long background-job rule, the
detailed edge cases.
