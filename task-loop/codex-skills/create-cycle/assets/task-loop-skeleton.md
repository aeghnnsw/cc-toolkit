## Runner Inputs

Each task runner must receive the task **seq**, **title**, GitHub **issue**
number, and branch name. If any is missing, stop before editing and ask the
task-loop controller for the missing value.

## General Rules

- **Tests first:** use `superpowers:test-driven-development`; "tested" means
  the Test conventions above.
- **Evidence before done:** use `superpowers:verification-before-completion`.
  Check a rubric item only when command output or artifact evidence proves it.
- **Pressure-test hard decisions:** use `dev-skills:pressure-test` for rubrics,
  plans, ambiguous requirements, failed approaches, and PR review.
- **Do the hard part:** failure and blocked outcomes require evidence that the
  central approach was attempted and alternatives were exhausted.
- **One reviewable PR:** if scope is too large, land the coherent core and record
  genuine follow-ups in Findings.
- **Use compute according to policy:** parallelize independent safe work and do
  not leave long jobs as unobserved foreground blockers.
- **Never violate invariants:** isolate in a worktree before edits; write only
  the assigned branch; end at an open PR; never merge from a worker task; stay
  inside the task and record follow-ups instead of acting on them.

## The Cycle

Do these steps once, in order, for one task.

### 1. Anchor

Read `docs/task-loop/directions.md`, then the source-of-truth docs above.
Confirm the task scope against its issue.

### 2. Set Up A Worktree

Start from the shared checkout and create a task-specific worktree for the
assigned branch. Retry transient git lock failures; a lock is not fatal.

```bash
ROOT="$(git rev-parse --show-toplevel)"
WT_PARENT="$(dirname "$ROOT")/.task-loop-worktrees"
mkdir -p "$WT_PARENT"
WT="$WT_PARENT/<seq>"
BR="<branch>"
n=0
until git -C "$ROOT" fetch origin || [ "$n" -ge 5 ]; do n=$((n + 1)); sleep 2; done
git -C "$ROOT" worktree remove --force "$WT" 2>/dev/null || true
git -C "$ROOT" worktree prune
n=0
until git -C "$ROOT" worktree add -B "$BR" "$WT" "origin/{{DEFAULT_BRANCH}}" || [ "$n" -ge 5 ]; do
  n=$((n + 1))
  sleep 2
done
cd "$WT"
```

Verify `git rev-parse --show-toplevel` equals `$WT`. If worktree setup fails,
stop before editing and report the failure.

### 3. Define The Rubric

Open `docs/task-loop/logs/<NNN>_<task>.md` where `<NNN>` is the zero-padded
task seq. Use `dev-skills:goal-rubric` to define binary acceptance criteria,
then pressure-test the rubric. Record the rubric in the study log and post it
to the issue as acceptance criteria.

### 4. Spec, Plan, Implement

Use `superpowers:brainstorming` to design the task. If the runner is
unattended or cannot ask ordinary design questions, infer from the proposal and issue, record assumptions, and use
`dev-skills:pressure-test` rather than blocking on ordinary design questions.

Use `superpowers:writing-plans` for the implementation plan. Review the plan
with `dev-skills:pressure-test`. Implement with
`superpowers:test-driven-development`.

### 5. Check The Rubric

Run every rubric item and capture the evidence in the study log. Fix until the
rubric is green, or record an honest failed/blocked outcome with evidence.

### 6. Reconcile

Compare results to the proposal and plan. If results invalidate a planning
assumption, do not silently edit the proposal. Record the finding so the
controller can decide any aim or roadmap revision.

### 7. Refresh Docs

Use `dev-skills:doc-update` for docs touched by the change. Commit doc updates
with the implementation.

### 8. Open The PR And Review

Finalize the study-log record in this shape:

```markdown
## Study log - task <seq>: <title>

**Outcome:** success

### Rubric
- [x] <binary acceptance item> - <command or artifact proving it>

### Evidence
<commands run and relevant output>

### Findings
<follow-up work, surprises, risks, or blockers>
```

`**Outcome:**` must be exactly one of `success`, `failed`, or `blocked`.
Rubric items are checked only when evidence proves them. `failed` and
`blocked` are last resorts backed by Findings.

Open a PR with this body shape:

```markdown
Refs #<issue>

**Outcome:** success
Study log: docs/task-loop/logs/<NNN>_<task>.md
```

Use `Refs #<issue>`, not `Closes #<issue>`. The controller owns issue closure.
Review the diff adversarially with `dev-skills:pressure-test`, fix or rebut each
substantive finding, and leave the PR open.

### 9. Report And Idle

Report the task seq, PR number and URL, head SHA, and declared Outcome. Do not
merge.

## Background Long Jobs

If a command may run for a long time, launch it in the background with a
run-unique log and status file:

```bash
J="job-<label>-<seq>-r<N>"
rm -f "$J.status"
( cmd >"$J.log" 2>&1; rc=$?; printf '%s\n' "$rc" >"$J.status"; exit "$rc" ) &
```

Before relying on the result, verify both conditions:

1. `<J>.status` exists and is exactly `0`.
2. `<J>.log` has been inspected for failure evidence such as `FAILED`, `Error`,
   `Traceback`, `Killed`, `OOM`, or partial output.

Never infer success from silence.

## Edge Cases

- **Cannot create a worktree:** stop before edits and report the failure.
- **Rubric cannot go green:** pressure-test alternatives, record evidence, and
  declare `failed` only after real attempts.
- **A dependency must land first:** declare `blocked` and name the blocker.
- **Human-only blocker:** escalate rather than guessing.
