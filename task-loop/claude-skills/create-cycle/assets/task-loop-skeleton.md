# Task-loop — project cycle & parameters

This file is the **cycle a task-loop worker follows for this project**. A worker reads
`docs/task-loop/directions.md` (human steering) first, then **follows this file strictly, step by
step** — building a TodoWrite list from the steps below and working them in order. The `cycle-worker`
agent contract holds the same general rules + invariants; they are **repeated here on purpose** so
this cycle is self-contained and every rule lands in the worker's todo list.

Generated by `create-cycle`; the `{{...}}` placeholders are this project's specifics. **After
upgrading the task-loop plugin, re-run `create-cycle`** to regenerate this file against the current
template.

## General rules (always — also in the agent contract; never violate the invariants)

- **Tests first** — `superpowers:test-driven-development`; "tested" = the **Test conventions** above.
- **Evidence before done** — `superpowers:verification-before-completion`: check a rubric item only
  when a command was run and its output confirms it.
- **Deliberate, don't stall** — at hard decisions (rubric, plan, PR review) use
  `dev-skills:discuss-with-codex`; record dispositions in the PR.
- **Don't skip the heavy lifting** — attack the hard part head-on; **exhaust approaches**
  (`discuss-with-codex`) before declaring `blocked`/`failed`. Those are **last resorts backed by
  evidence** (study-log **Findings**), never an escape from difficulty or a swap for an easier deliverable.
- **One reviewable PR** — if oversized, do the coherent core (**including the hard, central part**) and
  leave genuine extras as study-log **Findings**.
- **Never foreground-block a long job** — see *Background long jobs*.
- **Use the compute you have** per the **Compute policy** — never single-thread parallelizable work
  (`Workflow`/inline subagents, never a sub-team).
- **Invariants (never violate):** isolate in your own worktree before any edit; write only your given
  branch; end at an open PR (**never `gh pr merge`**); stay inside your task — report
  follow-ups/plan-effects in the PR, never act on them.

## The cycle (do once, in order, for your one task)

### 1. Anchor
Read `docs/task-loop/directions.md` (steering, highest priority after the invariants), then the
source-of-truth docs above. Confirm the task scope against its issue.

### 2. Set up your worktree
You start in the orchestrator's **shared** checkout, with several workers running against it at once,
so create and enter your **own** worktree for your branch, keyed on your `seq`, branched from the
default branch. Git is shared, so retry transient `*.lock`s — a lock is never fatal:

```bash
ROOT="$(git rev-parse --show-toplevel)"
WT_PARENT="$(dirname "$ROOT")/.task-loop-worktrees"; mkdir -p "$WT_PARENT"
WT="$WT_PARENT/<seq>"; BR="<branch>"        # <branch> from the spawn prompt (prefix per Branch prefixes)
n=0; until git -C "$ROOT" fetch origin || [ $n -ge 5 ]; do n=$((n+1)); sleep 2; done
git -C "$ROOT" worktree remove --force "$WT" 2>/dev/null; git -C "$ROOT" worktree prune   # fresh start
n=0; until git -C "$ROOT" worktree add -B "$BR" "$WT" "origin/{{DEFAULT_BRANCH}}" || [ $n -ge 5 ]; do
  n=$((n+1)); sleep 2; done
cd "$WT"
```

**Verify** `git rev-parse --show-toplevel` == `$WT` (≠ the shared root). `$WT` is your working root for
the whole cycle — prefix repo-touching shell with `cd "$WT"` (or `git -C "$WT" …`) since cwd may not
persist across tool calls. If you cannot create/enter a worktree after the retries, **report that to
the orchestrator and stop** — no edits.

### 3. Define the rubric (binary)
Open your study-log record `docs/task-loop/logs/<NNN>_<task>.md` (`<NNN>` = your `seq`, zero-padded —
this is your only index). Use `dev-skills:goal-rubric` for a binary pass/fail rubric (each item a test
name, a numeric tolerance, a gate verdict, or an artifact path); **finalize it with a
`discuss-with-codex` pass** — don't skip it even when it looks obvious. Write it into the **Rubric**
section of that record and post it to the issue as acceptance criteria.

### 4. Spec → plan → implement
- `superpowers:brainstorming` to design — but **decline its user-question and user-approval gates**
  (this runs autonomously); route open questions to `discuss-with-codex`. Write the spec to
  `docs/superpowers/specs/YYYY-MM-DD-<slug>-design.md`.
- `superpowers:writing-plans` → a step-by-step plan; review it with `discuss-with-codex`.
- Implement with `superpowers:test-driven-development` (tests first). Parallelize independent work per
  the **Compute policy**; long compute → background (see below).

### 5. Check the rubric
Run every rubric item and capture real output into the record's **Evidence**
(`superpowers:verification-before-completion`). Fix until green, or descope honestly — descopes become
**Findings**, never silent omissions.

### 6. Reconcile
Compare results to the proposal/plan. If a result **invalidates a plan assumption**, do **not** edit
`proposal.md` — record it in the record's **Findings**; the orchestrator decides any plan revision.

### 7. Refresh docs
Bring every doc the change touched to current truth (`dev-skills:doc-update`: README, docstrings, the
relevant source-of-truth sections). Route change history to `docs/CHANGELOG.md`. Commit doc updates
**with** the implementation.

### 8. Open the PR and review with Codex
**Finalize the study-log record** `docs/task-loop/logs/<NNN>_<task>.md` — **Outcome**
(`success` / `failed` / `blocked`), binary **Rubric**, **Evidence**, **Findings** — per
**`${CLAUDE_PLUGIN_ROOT}/references/pr-findings.md`**, and commit it **with** the work. Commit
attribution-free (write the message to a file, `git commit -F`; stage by explicit path). Push to your
branch and open the PR with an **explicit head**:
`gh pr create --head <branch> --base {{DEFAULT_BRANCH}} --body-file <body>`. Keep the PR body short:
`Refs #<issue>` + the **Outcome** line + a pointer to the record. **Review the PR adversarially with
`discuss-with-codex`** (feed it the real diff); treat it with `superpowers:receiving-code-review`
rigor; fix or rebut each point; re-review until no blocking issues.

### 9. Report and idle
Send the orchestrator **one line** — your `seq`, the PR number + URL + head SHA, and the declared
**Outcome** — then **go idle**. Do not merge and do not self-terminate. If the orchestrator sends a
shutdown request for that closed `seq`, approve it and exit gracefully.

## Background long jobs (never foreground-block)

If a shell command may approach the ~10-min foreground cap or has unbounded variance (full test
suites, builds, installs, large downloads, training) — **don't** run it as a blocking foreground call
(it would be killed at the cap and stall your turn). Run it with **`Bash` `run_in_background`**,
writing output + exit status to a **run-unique** stem you pick, and clear the status file first:

```bash
J=job-<label>-<seq>-r<N>     # bump <N> on every rerun
rm -f "$J.status"
( cmd > "$J.log" 2>&1; rc=$?; printf '%s\n' "$rc" > "$J.status"; exit "$rc" )
```

Then verify BOTH, in order: (a) `<J>.status` **exists** and is exactly `0` (a missing status = the run
was killed = failure, never success); (b) **`Read` `<J>.log`** and scan for failure evidence
(`FAILED`, `Error`, `Traceback`, `Killed`, `OOM`, partial output) — inspect each match in context.
Proceed only if the status is `0` **and** every match is benign. **Never infer success from silence.**
For streamed progress use **`Monitor`**; for completion signaled outside the process (a server
becoming ready, a remote job) use a `run_in_background` `until`-loop on that marker. A bounded retry
`sleep` (the git-lock loop above) is fine. Non-shell long calls (a `Workflow` run) use their own
bounded completion — never force them into this wrapper.

## Edge cases

- *Cannot create/enter a worktree* → report to the orchestrator and stop before any edit.
- *Rubric can't go green* → **exhaust approaches first** (iterate, `discuss-with-codex`); descope only
  as a last resort, honestly (descopes → study-log **Findings**, never silent omissions) and set the
  **Outcome** accordingly. A red rubric is a reason to try harder, not to lower the bar.
- *Something must land first* → set **Outcome: blocked** and name the blocker concretely in
  **Findings**; the orchestrator re-tasks it. Don't route around it.
- *Genuine human-only blocker* (compute/budget beyond this environment, missing licensed access, an
  irreversible out-of-band action, or scientific ambiguity the proposal + a `discuss-with-codex` round
  can't resolve) → escalate to the orchestrator rather than guessing.
